# Architecture and Component Boundaries

This document describes the current repository architecture for the Terraform plus Ansible Kubernetes deployment workflow.

## Repository Layers

The repository is organized into four practical layers.

### 1. Terraform provisioning layer

Terraform under [`terraform/aws/`](../terraform/aws/) and [`terraform/proxmox/`](../terraform/proxmox/) is responsible for:

- provisioning platform infrastructure
- exposing platform outputs
- rendering platform-specific Ansible inventory files from [`inventory.tpl`](../terraform/aws/inventory.tpl) and [`inventory.tpl`](../terraform/proxmox/inventory.tpl)

Terraform is not responsible for Kubernetes lifecycle actions after the hosts are provisioned.

### 2. Ansible cluster configuration layer

Ansible under [`ansible/`](../ansible/) is responsible for validating topology and converging hosts into a working Kubernetes cluster.

The playbook entrypoint is [`ansible/site.yml`](../ansible/site.yml). The repository currently does not use a `playbooks/` directory or a `site.yaml` entrypoint.

The playbook performs these major phases:

1. topology and inventory validation on localhost
2. baseline preparation on `cluster_nodes`
3. load balancer configuration on `load_balancers`
4. PKI staging on `etcd:control_plane`
5. external etcd deployment on `etcd`
6. bootstrap control-plane initialization on `control_plane[0]`
7. secondary control-plane joins on `control_plane[1:]`
8. worker joins on `workers`
9. CNI installation from `control_plane[0]`
10. post-deployment verification from `control_plane[0]`

### 3. Shell workflow layer

The operator-facing wrappers are:

- [`script/main.sh`](../script/main.sh): current primary wrapper
- [`Makefile`](../Makefile): convenience targets around the primary wrapper and targeted HAProxy commands
- [`setup-k8s.sh`](../setup-k8s.sh): compatibility wrapper with a smaller interface

These wrappers coordinate Terraform and Ansible, but they do not define cluster topology themselves. Topology is supplied by Terraform-generated or manually created inventory.

### 4. Generated and local artifact layer

The current automation writes and consumes runtime artifacts in several places:

- generated inventories under `ansible/inventories/<platform>/hosts.ini`
- local Ansible artifacts under the inventory-defined `local_artifact_dir`, defaulting to `{{ playbook_dir }}/.artifacts`
- kubeadm and join artifacts on nodes under `kubeadm_artifact_dir`, defaulting to `/etc/kubernetes/ansible-artifacts`
- backups under [`backups/`](../backups/)
- logs under [`logs/`](../logs/)

## Current Role Boundaries

The role structure under [`ansible/roles/`](../ansible/roles/) is intentionally separated by responsibility.

### [`ansible/roles/common/`](../ansible/roles/common/)

Host baseline preparation:

- swap handling
- firewall-related preparation
- sysctl and kernel module configuration
- package installation
- time synchronization and other host prerequisites

### [`ansible/roles/containerd/`](../ansible/roles/containerd/)

Container runtime preparation. The repository currently assumes containerd, not Docker.

### [`ansible/roles/k8s/`](../ansible/roles/k8s/)

Kubernetes package installation and host-level cluster prerequisites.

### [`ansible/roles/loadbalancer/`](../ansible/roles/loadbalancer/)

HAProxy configuration for routing the Kubernetes API endpoint to the control-plane backends.

This role is only applied to the canonical `load_balancers` inventory group and only when `loadbalancer_enabled=true`.

### [`ansible/roles/cert/`](../ansible/roles/cert/)

Certificate staging for the external-etcd topology:

- shared etcd CA generation
- apiserver-to-etcd client certificate generation
- per-node etcd server, peer, and healthcheck client certificate generation
- distribution of the required certificate material to etcd and control-plane nodes

### [`ansible/roles/etcd/`](../ansible/roles/etcd/)

Dedicated external etcd deployment, service configuration, and health verification.

This role is modeled for dedicated etcd hosts in the canonical `etcd` group.

### [`ansible/roles/control_plane/`](../ansible/roles/control_plane/)

kubeadm control-plane lifecycle:

- bootstrap initialization on the first control-plane node
- durable join command generation
- certificate upload for secondary control-plane nodes
- sequential join of additional control-plane nodes

### [`ansible/roles/worker/`](../ansible/roles/worker/)

Worker join execution using the durable join script generated by the bootstrap control-plane node.

### [`ansible/roles/cni/`](../ansible/roles/cni/)

Installs the selected cluster networking plugin after the control plane and workers have joined.

The sample group variables currently define `cni_plugin: calico`, while a Flannel manifest URL also exists as an available variable.

## Topology Contract

The topology contract is centered on canonical inventory groups:

- `bastion`
- `load_balancers`
- `etcd`
- `control_plane`
- `workers`
- `k8s_cluster`
- `cluster_nodes`
- `services`

The control flow in [`ansible/site.yml`](../ansible/site.yml) and validation in [`script/modules/ansible.sh`](../script/modules/ansible.sh) expect these names. Legacy names such as `masters` are not the Ansible contract, even though Terraform internals may still use legacy local variable names while rendering the canonical output.

See [`docs/inventory-topology.md`](inventory-topology.md) for the full contract.

## Control-Plane and etcd Model

The current repository state models:

- a kubeadm-managed HA control plane
- an optional HAProxy-based front door for the API endpoint
- dedicated external etcd nodes
- verification of etcd health from both etcd nodes and the bootstrap control-plane host

This matters operationally because:

- certificate generation differs from stacked-etcd kubeadm defaults
- health checks and backup workflows explicitly target the `etcd` group
- wrappers assume the first control-plane host is addressable as `control_plane[0]`

## Wrapper Responsibilities and Boundaries

### [`script/main.sh`](../script/main.sh)

Current primary workflow wrapper.

Responsibilities:

- parse user actions and options
- validate environment and project structure
- run Terraform workflow operations per platform
- validate generated inventory and Ansible structure
- invoke [`ansible/site.yml`](../ansible/site.yml) or another specified playbook
- run health, backup, update, troubleshooting, and maintenance helpers

### [`Makefile`](../Makefile)

Human-friendly shortcuts for repeatable operations.

Responsibilities:

- common platform workflows like `*-plan`, `*-apply`, `*-ansible`, `*-health`
- targeted HAProxy management against `load_balancers`
- setup helpers for example Terraform variable files

### [`setup-k8s.sh`](../setup-k8s.sh)

Legacy simplified wrapper.

Responsibilities:

- basic `plan`, `apply`, `destroy`, and `ansible` flows
- lightweight dependency checking

It does not expose the broader workflow surface implemented by [`script/main.sh`](../script/main.sh).

## Runtime Assumptions Reflected in Code

The current codebase assumes:

- Linux hosts reachable through SSH
- containerd as the runtime via the `containerd` role
- kubeadm as the cluster bootstrap tool
- dedicated external etcd nodes when `external_etcd_enabled=true`
- at least one worker node in the intended topology
- at least one control-plane node, with sequential expansion for HA control planes
- a defined `control_plane_endpoint` for kubeadm configuration
